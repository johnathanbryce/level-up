## Scope When Ready

### Architecture Teardowns

- [ ] Study 2-3 real-world architectures (from engineering blogs: Uber, Stripe, Discord, etc.)
- [ ] For each: identify components, trade-offs made, what you'd do differently
- [ ] Practice explaining architectures to Claude as if in an interview

### Build vs Buy Decisions

- [ ] Framework for evaluating build vs buy (cost, control, maintenance, time-to-market)
- [ ] Case studies: when companies built custom vs used off-the-shelf
- [ ] Common traps: NIH syndrome, vendor lock-in, hidden maintenance costs

### Trade-Off Analysis

- [ ] Speed vs correctness vs maintainability — situational framework
- [ ] Abstraction benefits vs long-term costs (leaky abstractions)
- [ ] Choosing boring technology — when innovation is risk, not reward
- [ ] Scope management: saying no to features, scoping for impact

### Open-Source Codebase Exploration

- [ ] Read through 1-2 well-structured open-source projects
- [ ] Identify patterns: how they organize code, handle errors, manage state
- [ ] Note what makes the codebase readable vs confusing

### Incident Analysis

- [ ] Read 3-5 public post-mortems
- [ ] For each: root cause, detection, response, prevention
- [ ] Pattern recognition: what types of failures recur across companies
- [ ] What would you have done differently?

## Definition of Done

Can analyze an architecture and identify strengths, weaknesses, and trade-offs. Can make and defend build vs buy decisions. Can read unfamiliar code and understand its structure. Has developed a framework for making engineering decisions under constraints.
