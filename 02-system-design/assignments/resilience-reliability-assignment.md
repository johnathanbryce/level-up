Scenario — you're designing a news aggregator (think Reddit / Hacker News). Homepage shows:

A ranked list of top stories — generated by your ranking service that scores posts using engagement + recency + user preferences.
Each story shows: title, author, vote count, comment count, and a thumbnail image (fetched from an external CDN).
A sidebar shows "Trending Now" — computed by a separate trending-analytics service.
A small widget shows the user's karma / notifications — fetched from your user service.

It's 2am. Your trending-analytics service has crashed. Your image CDN is timing out 80% of requests. Everything else is healthy.

Three questions:

1. What is the critical path for a user visiting your homepage? (What MUST render for the user to get value?)
2. How do you degrade each of the 4 components above so the page still loads and is useful?
3. What specifically should NOT happen — what's the anti-pattern you're protecting against?

# My Answer

1. Critical path is stories / curated stories -> per-story-data -> user karma/notifications -> trending now
   - Reasoning: Stories MUST be shown to the user. Preference for curated stories but if this service is down, show the chronological story feed. If both services are down, serve cached stories. For per-story-data this goes hand-in-hand with the stories service preference hierarchy as well. If per-story-data is down, once again serve cached data so a user sees something. Karma and notifications is the third most important as users can get by and still use the app without these. If this service is down, there are many ways to deal with this but you could show an error icon / flag over this area on the UI indicating "the service will be back up ASAP" and if necessary show them cached karma and notifications. Trending now is the least critical and can just be replaced with either a cache and a notif that the service is down or just an updated indicating the service is down and here is XYZ in its place (could be cached prior trending articles)

2. This is mentioned in answer 1 per service. Summary:
   - Fall back to defaults for stories if curated stories is down
   - Fall back to cached data for per-story data if per-story-data is down
   - Disable or cache non critical features like karma/notifs and ternding now

3. Anti pattern is to avoid all or nothing failures. If an non-critical feature like Trending Now is down, there is no need to crash the entire app. Just gracefully handle this section of the UI

## Better answer

### Q1 — Critical path

- **Critical (MUST render):** Ranked list of stories with core metadata — title, author, vote count, comment count. That's the product. A user who sees clickable stories with vote counts has gotten the value.
- **Partially-critical (try to render, OK to degrade):** Thumbnails. Improve UX but page is still useful without.
- **Non-critical (fine to hide entirely):** Trending sidebar, karma/notifications widget.

### Q2 — What happens to each component in THIS scenario

1. **Ranked list of stories (healthy)** — renders normally from ranking service. Critical path, unchanged.
2. **Story metadata — title, author, votes, comments (healthy)** — renders normally alongside each story.
3. **Thumbnails (CDN timing out 80%)** — don't block story rendering on them:
   - Render the list of stories immediately with text + placeholder divs where images go.
   - Each thumbnail request gets a short timeout (~200ms) — if CDN doesn't respond, show a default placeholder (gray box, generic icon, CSS-only text card).
   - Lazy-load images below the fold so only visible thumbnails are attempted.
   - Consider a secondary image source as failover if available.
   - **Page renders fully and becomes interactive even if every thumbnail fails.**
4. **Trending sidebar (service crashed)** — detect via circuit breaker, hide the sidebar entirely OR show last-known-good cached trending with a "updated X minutes ago" timestamp. Do NOT show a broken/empty widget with a spinner.
5. **Karma/notifications widget (healthy)** — renders normally. But because it's non-critical, **even if it were down, the page should still render** — the widget just wouldn't appear.

### Q3 — The anti-pattern

Specific anti-pattern: **the homepage blocks rendering on any single failing dependency.**

Concrete examples of what this would look like:

- Server-side rendering waits for the trending-analytics service before sending any HTML → request hangs 30 seconds → user sees spinner → concludes site is down.
- Each story HTTP-request-blocks on its thumbnail → 80% timing out at 30s each → page takes forever → user bounces.
- Homepage waits for user service karma call before rendering anything → user service times out → blank page.

Rule: **render the critical path first, enhance with non-critical data when/if it loads.** Never let an optional component block a required one.

### Interview vocabulary

- **"Non-blocking critical path"** — core render path doesn't wait for optional data.
- **"Graceful fallback per component"** — each non-critical component has its own fallback (hide / cache / default).
- **"Timeout-and-placeholder"** — the pattern for the thumbnail case specifically.
